# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Workout Timer ‚Äî a Flutter Android app (single platform; iOS/web are not configured) for building and running interval workouts with TTS announcements. This is a fork of `blockbasti/just_another_workout_timer` (published on F-Droid and upstream GitHub Releases); the fork is published only via this repo's GitHub Releases under a different `applicationId` (`com.craeckie.workouttimer`) and therefore installs side-by-side with the upstream app ‚Äî backups can be moved between them via export/import.

## Commands

Dev workflow uses [`script_runner`](https://pub.dev/packages/script_runner) (the `scr` command). Install once with `dart pub global activate script_runner`.

- `flutter pub get` ‚Äî fetch dependencies
- `scr build` ‚Äî runs the full codegen pipeline: `licenses`, `translations` (intl_utils), `launcher_icons`, `native_splash`, `build_runner`. Individual steps are also available (`scr translations`, `scr build_runner`, etc. ‚Äî see `pubspec.yaml` `script_runner:` block).
- `flutter build apk` ‚Äî build release APK (debug APK is what CI publishes)
- `flutter test` ‚Äî run unit tests
- `flutter test test/utils_test.dart` ‚Äî single file; append ` --plain-name "Utils.formatSeconds zero"` for one test
- `flutter analyze` ‚Äî lint. `lib/layouts/oss_license_page.dart` needs `scr licenses` to generate `lib/oss_licenses.dart` first.

Codegen outputs (`lib/generated/**`, `lib/**/*.g.dart`, `lib/oss_licenses.dart`) are excluded from analyzer and should not be hand-edited ‚Äî re-run `scr build` instead.

### `workout.g.dart` quirk

`lib/utils/workout.dart` declares `part '../workout.g.dart'`, so the committed generated file lives at `lib/workout.g.dart` ‚Äî not next to its source. `build_runner`'s combining-builder writes to `lib/utils/workout.g.dart` (which gets ignored) and emits a `must be included as a part directive` warning every run, so the committed file does **not** auto-update after `scr build_runner`. When you add a `@JsonSerializable` class, copy the new functions from `.dart_tool/build/generated/just_another_workout_timer/lib/utils/workout.json_serializable.g.part` into `lib/workout.g.dart` by hand.

## Architecture

Three-layer split inside `lib/`:

- **`utils/workout.dart`** ‚Äî pure data model (`Workout` ‚Üí `Set` ‚Üí `Exercise`, plus `Backup`). All JSON via `json_serializable` codegen (`workout.g.dart`). `Workout.fileVersion` drives migration; bump it when changing the schema and add a case to `Migrations._migrate`.
- **`utils/storage_helper.dart` + `utils/migrations.dart`** ‚Äî file I/O. Workouts live as individual JSON files under `getExternalStorageDirectory()/workouts/`. `Migrations.runMigrations()` is called from `main()` on startup and mutates files in place. Filenames go through `Utils.removeSpecialChar` for safety.
- **`utils/history_helper.dart`** ‚Äî completion-log I/O. Appends a `HistoryEntry` (title + timestamp) to a single `getExternalStorageDirectory()/history.json` file. Sibling of `workouts/`, deliberately outside it so `getAllWorkouts()` and backup export ignore it. The single write site is `Timetable.buildTimetable()`'s "workout complete" callback ‚Äî partial runs are not recorded.
- **`layouts/`** ‚Äî UI screens: `home_page` (list), `workout_builder` (edit), `workout_runner` (run), `history_page` (past completions, swipe-to-delete + clear-all), `settings_page`, `oss_license_page`. The runner delegates timing to `utils/timetable.dart`.

`Timetable` (`utils/timetable.dart`) is the timer engine: a `ChangeNotifier` that holds a `SplayTreeMap<int, Function>` of scheduled callbacks (TTS announcements, beeps, exercise transitions) and ticks them on a `Timer.periodic`. It currently couples `BuildContext`, the `Prefs` singleton, and static `TTSHelper`/`SoundHelper`, which is why it has no unit tests ‚Äî refactor before testing.

### Preferences ‚Äî two libraries coexist

`main.dart` initializes **both** `prefs` (`Prefs.init()`, used elsewhere as `Prefs.getX('key')`) and `pref` (`PrefServiceShared` wrapped in a `PrefService` widget, used for settings UI). When adding a preference, match whichever the surrounding code uses; do not assume one canonical API.

### Localization

Strings live in `lib/l10n/*.arb`; `lib/generated/l10n.dart` (the `S` class) is generated by `scr translations` (intl_utils). Translations are managed externally via Weblate ‚Äî edit `.arb` files, not the generated `S` class.

### Theming

Material 3 with `dynamic_color` (Material You). The app reseeds from `lightDynamic`/`darkDynamic` when available, falling back to `Colors.blue`. `flutter_phoenix` is wrapped around the app so settings changes can hot-restart it.

## Lint conventions (`analysis_options.yaml`)

`require_trailing_commas` and `prefer_expression_function_bodies` are enforced ‚Äî both affect formatting. `sort_pub_dependencies` is on, so keep `pubspec.yaml` alphabetized within each section.

## Commit policy

- Commit after every successful change ‚Äî do not batch unrelated work.
- **Always amend** the previous commit when the new change is a continuation of it (fixup, follow-up tweak, missed file, related polish). Reserve new commits for genuinely separate changes.
- Exception: if the previous commit has already been pushed to the remote, never amend it ‚Äî make a new commit instead.

## Android toolchain

Current pinned versions (in `android/`):

- AGP `8.7.0` (`settings.gradle`) ‚Äî minimum 8.6.0 required by current `androidx.core` transitive deps.
- Gradle wrapper `8.9` (`gradle/wrapper/gradle-wrapper.properties`) ‚Äî AGP 8.7 requires ‚â• 8.9.
- Kotlin Gradle plugin `2.2.0` (`settings.gradle`) ‚Äî matches the metadata version used by `share_plus 12.x`.
- `compileSdk 36` (`app/build.gradle`) ‚Äî required by `fluttertoast 9.x`.
- `org.gradle.jvmargs=-Xmx2048M` with daemon/parallel/workers disabled in `gradle.properties` ‚Äî host-specific tuning for a ~4 GiB build VM. Bump back up and re-enable the daemon on a workstation with more headroom.
- Release signing falls back to debug-signing when `key.properties` is absent. `release.sh` re-signs the unsigned-by-the-build APK with `apksigner` afterward ‚Äî that flow still works because `apksigner` doesn't care what signature is on the input.

### Vendored `soundpool`

`soundpool` is discontinued upstream; its only published version (2.4.1) still references the V1-embedding `PluginRegistry.Registrar` and won't compile on Flutter 3.32+. A patched copy lives in `local_plugins/soundpool/` (V1 `registerWith` companion stripped; V2 `FlutterPlugin` path untouched) and is wired in via `dependency_overrides` in `pubspec.yaml`. If you ever bump the audio plugin, you can drop both the override and the vendored directory.

## Release

Releases are cut by `release-please` on push to `main`.

1. Land commits on `main` using conventional-commit prefixes: `feat:` ‚Üí minor
   bump, `fix:` ‚Üí patch bump, `chore:`/`docs:`/`refactor:`/`test:` ‚Üí no bump.
   Breaking changes go in a `BREAKING CHANGE:` footer and trigger a major bump.
2. `release-please` opens (or updates) a "release PR" that bumps
   `pubspec.yaml: version:` and `CHANGELOG.md`. Review and merge it when you
   want to ship.
3. Merging the release PR triggers `.github/workflows/release.yml`:
   `scr build` ‚Üí `flutter analyze` ‚Üí `flutter test` ‚Üí
   `flutter build apk --split-per-abi --target-platform android-arm64,android-arm`
   (no x86_64) ‚Üí signed via `r0adkll/sign-android-release@v1` using the
   repo secrets `SIGNING_KEY` (base64-encoded keystore) and
   `SIGNING_KEY_PASSWORD` ‚Üí each APK uploaded to the GitHub Release as
   `com.craeckie.workouttimer-<abi>.apk`. `release.sh` (local-dev) builds
   arm64-only via `--target-platform android-arm64`.
4. CI on every other push (`CI.yml`) runs the same analyze + test + build
   steps but produces an unsigned debug APK artifact for smoke-testing
   branches before merge.

Never bump `pubspec.yaml: version:` by hand ‚Äî `release-please` owns it.
Never `--amend` a commit that has already been pushed; `release-please` walks
the commit log, and rewriting history confuses it.

### Signing keystore (one-time setup per fork)

```
keytool -genkeypair -v -keystore release.jks -keyalg RSA -keysize 2048 \
        -validity 10000 -alias key
base64 -w0 release.jks      # paste into the SIGNING_KEY repo secret
                            # store the keystore password in SIGNING_KEY_PASSWORD
```

Both the keystore alias (hard-coded as `key` in `release.yml`) and the
`r0adkll/sign-android-release` action use a single password for both the
keystore and the key ‚Äî generate the keystore accordingly.

`release.sh` is the local-dev equivalent and re-signs the build output with
`apksigner` from a checked-out keystore (`../my-debug.jks`); it is not used
by CI.
